## Context

The workspace root is not a git repository. It contains five independent child repositories and several root-level local artifacts:

- `mist`: NestJS backend monorepo with OpenSpec state, generated `dist`, `node_modules`, and ignored test output.
- `mist-fe`: Next.js frontend with tracked fixtures/results synchronized from the backend.
- `mist-datasource`: FastAPI datasource bridge with local `.venv`, Python caches, deployment scripts, and modified Windows service scripts.
- `mist-skills`: Python package for agent skills.
- `mist-deploy`: deployment-script repository with an active linked worktree at `.worktrees/mist-deploy-tdx-guard`.

The cleanup must protect existing user work. Current known dirty state includes untracked OpenSpec/planning artifacts in `mist` and modified scripts in `mist-datasource`. These must not be reverted or overwritten.

## Goals / Non-Goals

**Goals:**
- Establish a repeatable audit process for each repository and root-level artifact.
- Delete local generated artifacts and stale scratch directories that are not part of any repository contract.
- Remove or ignore recurrence sources for generated artifacts where safe.
- Identify tracked files that are likely obsolete, then remove only when references and tests support removal.
- Leave each child repository with clear git status and a concise cleanup report.

**Non-Goals:**
- Do not rewrite product architecture or rename active modules as part of this cleanup.
- Do not archive or complete unrelated OpenSpec changes.
- Do not delete clean linked worktrees that still represent active branches unless explicitly approved.
- Do not delete tracked test fixtures merely because they are generated; fixtures/results are kept when scripts or tests consume them.

## Decisions

### Decision 1: Treat each child repository independently

Each repository has its own git history, ignore rules, tests, and release surface. Cleanup actions are evaluated inside that repository first, then checked for cross-repository references from the workspace root.

Alternative considered: create a root-level cleanup as if the workspace were a monorepo. This is rejected because the root has no `.git` and root artifacts are not commit candidates for any child repo.

### Decision 2: Use four cleanup classes

Every candidate is classified before removal:

1. **Keep tracked**: source, tests, docs, fixtures, config, lockfiles, CI, deployment files, and specs used by current workflows.
2. **Ignore or leave local**: dependency directories, build output, caches, editor state, assistant-local config, and machine-specific env files.
3. **Delete local artifact**: ignored or untracked files that are reproducible and not referenced by current commands.
4. **Tracked removal candidate**: tracked file or business logic with no current imports, no script references, and no documented workflow dependency.

Tracked removal candidates require extra evidence: reference search, relevant tests, and a small diff. If evidence is weak, keep the file and record why.

### Decision 3: Prefer prevention over repeated cleanup

When a removable local artifact is generated by a normal command, update the relevant `.gitignore` or test cleanup behavior. Example: `mist-datasource/scripts/test_windows_scripts.ps1` creates `../old-service-wrapper/old-service-wrapper.exe`; the root `old-service-wrapper/` directory is local test output and should either be cleaned by the test or documented as non-commit local output.

Alternative considered: only delete current artifacts. This leaves the same clutter to reappear after the next test or build.

### Decision 4: Use OpenSpec as the audit ledger

The OpenSpec change tracks the cleanup policy and task checklist. Implementation evidence can be captured in task notes or a short report, but the code changes remain in the relevant child repositories.

## Risks / Trade-offs

- **Risk: deleting a file used only on Windows** -> Mitigation: search deployment scripts and docs before removing Windows-specific assets; avoid deleting scripts with unclear runtime dependency.
- **Risk: deleting fixtures that look generated but are part of tests** -> Mitigation: keep fixture/result files referenced by sync scripts, test imports, or docs.
- **Risk: existing dirty user changes get mixed with cleanup** -> Mitigation: snapshot git status before each repo cleanup and keep diffs scoped.
- **Risk: active worktree removed accidentally** -> Mitigation: use `git worktree list` and only remove non-git scratch folders automatically.
- **Risk: static analysis misses dynamic imports** -> Mitigation: combine import/reference search with targeted test/build commands; keep uncertain candidates.

## Migration Plan

1. Capture a repository inventory: tracked, untracked, ignored, dirty status, and root-level artifacts.
2. Delete obvious local artifacts: `.DS_Store`, generated caches, local dummy `old-service-wrapper/` copies, and non-git scratch worktree remnants.
3. Update ignore/test cleanup rules for recurring artifacts.
4. Analyze tracked stale candidates by references and current workflow usage.
5. Remove proven-unused tracked files or logic in small commits/diffs per repository.
6. Run verification commands for affected repositories.

Rollback is standard git revert for tracked changes. Local artifact deletions are reproducible via install/build/test commands; if a non-reproducible local artifact is discovered, restore it from backup or stop before deletion.

## Open Questions

- Whether active but old OpenSpec changes should be archived, kept, or deleted is a product decision and is not automatic cleanup.
- `.trae` historical docs in `mist-datasource` have no external references after current documentation ownership checks, so they are removed as obsolete tool-process artifacts.
